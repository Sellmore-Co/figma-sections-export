Figma Design Framework Specification

A complete guide for structuring Figma files and exporting sections as Liquid + CSS for the next-campaign-page-kit framework.

0. Getting Started

Follow these steps to set up your environment from scratch. You need three things working together: this repo, the campaign-page-kit, and the Figma MCP plugin inside Claude Code.

Prerequisites

  • Node.js 18+ — nodejs.org
  • Claude Code CLI — claude.ai/code
  • Figma MCP plugin enabled in Claude Code (see Step 2)

Step 1 — Clone and install

git clone <this-repo>
cd figma-templates-export
npm install

npm install pulls next-campaign-page-kit from npm and makes the campaign-dev, campaign-build commands available.

Step 2 — Add your Figma personal access token

cp .env.example .env

Open .env and paste your token. This is used by the save-ref.sh script to download reference screenshots of each section at each breakpoint via the Figma REST API.

Getting your token

Figma → Account Settings → Personal Access Tokens → Generate new token

Scopes needed: File content (read) + File metadata (read). The token is stored only in your local .env file which is gitignored.

Step 3 — Enable the Figma MCP plugin in Claude Code

The Figma MCP server gives Claude direct read access to your Figma files — no manual copy-paste of node data needed. Claude can fetch design context, screenshots, and asset URLs directly.

Open Claude Code and run:

claude mcp add

Or add it manually to your MCP config at ~/.claude/claude_desktop_config.json:

{
  "mcpServers": {
    "figma": {
      "command": "npx",
      "args": ["-y", "@figma/mcp-server"],
      "env": {
        "FIGMA_ACCESS_TOKEN": "your-figma-personal-access-token"
      }
    }
  }
}

Getting your Figma access token

Figma → top-left menu → Account Settings → scroll to Personal Access Tokens → Generate new token. Give it a name and copy the token immediately — it won't be shown again.

Restart Claude Code after adding the plugin. You'll know it's active when Claude can call get_design_context and return live screenshots from a Figma URL without you providing any extra data.

Step 3 — Get the Figma links for a responsive section

Every section has 3 variant frames in Figma: desktop, tablet, and mobile. The easiest way to reference them is by copying the link directly from Figma — no need to manually find node IDs.

Cmd+L — the fastest way

Click a frame on the Figma canvas, then press Cmd+L (Mac) or Ctrl+L (Windows), or right-click → Copy link to selection. This copies a URL with the node ID already embedded. Paste it directly into Claude.

The copied URL looks like:

https://www.figma.com/design/ia7650Y3lLte4WVYARNvSX/Debranded-Sections?node-id=143-10518

Repeat for the tablet and mobile variant frames. You'll have 3 URLs total.

Step 4 — Start an export

Open Claude Code in the figma-templates-export directory and paste all 3 Figma links in one message:

Export this section:
https://www.figma.com/design/{fileKey}/...?node-id=143-10518
https://www.figma.com/design/{fileKey}/...?node-id=143-10610
https://www.figma.com/design/{fileKey}/...?node-id=143-12936

Claude fetches all 3 breakpoints in parallel, saves reference screenshots, generates the Liquid partial, and downloads all assets. See Section 13 for the full step-by-step process.

Step 5 — Preview

npm run dev

Select your campaign from the list → browser opens at http://localhost:3000/{campaign-slug}/ with live reload on file save.

Framework Reference

1. Introduction

This specification defines how to structure a Figma design system so that an external script (using the Figma REST API) can reliably extract sections and convert them into production-ready code that plugs directly into the next-campaign-page-kit framework.

The Core Principle

Every Figma frame, component property, style, and naming convention maps 1:1 to a framework concept. The designer's work IS the developer spec. No interpretation needed.

What the Framework Expects

The next-campaign-page-kit is a LiquidJS-based static site generator for e-commerce campaign funnels (presale, checkout, upsell, receipt pages). Each campaign is fully isolated with its own layouts, includes, assets, and configuration. Key conventions:

  • Templates use LiquidJS (not Shopify Liquid — nearly identical but a few edge cases differ)
  • Three custom template helpers: campaign_asset, campaign_link, campaign_include
  • Assets are copied as-is (no build step by default) — CSS can be Tailwind CDN, custom, or both
  • Sections live in _includes/ as HTML partials
  • Pages use YAML frontmatter to declare metadata, styles, and scripts
  • Campaign data comes from campaigns.json and is accessible via {{ campaign.* }}

How the Export Script Works

The export script reads the Figma file via the REST API and uses structured metadata (component names, properties, descriptions, plugin data) to generate framework-compatible output. The script must:

  1. Traverse the Figma node tree, identifying section-level frames by naming convention
  2. Read component properties to generate Liquid template variables
  3. Map Auto Layout properties to CSS (Tailwind classes or custom CSS)
  4. Export image fills as files and wrap references in campaign_asset
  5. Generate the Liquid partial with conditional blocks, loops, and variable interpolation
  6. Optionally generate page-level files with correct frontmatter

4. Figma File Structure

Organize your Figma file with these pages. The naming is prescriptive — the export script relies on it.

📁 Figma File: "Campaign Design System"
├── 📄 Tokens — Colors, typography, spacing, effects
├── 📄 Components — Atomic/reusable components (buttons, badges, inputs, etc.)
├── 📄 Sections — Exportable section components (hero, features, CTA, etc.)
├── 📄 Pages — Assembled page compositions (presale, checkout, upsell, receipt)
└── 📄 Sandbox — Experiments, explorations (not exported)

Rule: Only the "Sections" page is exported

The export script scans only the Sections page for exportable frames. Components on the Components page are used via instance references. The Pages page is reference-only for page assembly order and frontmatter metadata.

9. Design Tokens: Typography

Typography Variables

Font sizes come from Figma Variables (--font/size-*), not Figma Text Styles. Each variable has a different fallback px value per breakpoint node — reading all three fallbacks is how you determine the responsive class.

How get_design_context surfaces this:

text-[length:var(--font/size-heading2,40px)]   ← desktop node fallback
text-[length:var(--font/size-heading2,32px)]   ← tablet node fallback
text-[length:var(--font/size-heading2,28px)]   ← mobile node fallback

Read the three fallback values across all three breakpoint nodes, then generate responsive Tailwind classes.

Figma VariableMobileTabletDesktopTailwind output
--font/size-heading132px36px44pxtext-[32px] md:text-[36px] lg:text-[44px]
--font/size-heading228px32px40pxtext-[28px] md:text-[32px] lg:text-[40px]
--font/size-heading324px26px28pxtext-[24px] md:text-[26px] lg:text-[28px]
--font/size-p-large20px20px24pxtext-[20px] lg:text-[24px]
--font/size-p-big18px20px20pxtext-[18px] md:text-[20px]
--font/size-p-medium16px18px18pxtext-[16px] md:text-[18px]
--font/size-p-small14px14px16pxtext-[14px] lg:text-[16px]
--font/size-p-tiny12px12px14pxtext-[12px] lg:text-[14px]

Breakpoint prefix rules

  • Only add md: / lg: prefixes where the fallback value actually changes between breakpoints.
  • If a size is the same across all three breakpoints, emit a single class with no prefix.
  • p-big: tablet AND desktop are both 20px → use md: not lg:
  • p-small: tablet stays at 14px → use lg: not md:
  • Read fallback values from section content nodes, not from shared components (e.g. the CTA button always outputs desktop fallbacks regardless of breakpoint).

Font Family Convention

Font weight and family are inlined per element — there is no shared text style. Read them directly from the MCP output (e.g. font-bold, font-semibold). Define the font family as a Figma variable. Store in campaigns.json as {{ campaign.font_family }}.

12. Variants & Responsive Design

Breakpoint Variants

Every section should have a breakpoint variant property with at minimum two values:

Variant ValueCSS BreakpointFigma Frame WidthRole
mobileDefault (no prefix)375pxBase mobile-first layout
tabletmd: (768px+)768pxTablet adjustments Optional
desktoplg: (1024px+)1440pxFull desktop layout

How Variants Map to CSS

The export script compares the mobile and desktop variants and generates responsive utility classes:

<!-- If mobile=vertical, desktop=horizontal layout -->
<div class="flex flex-col lg:flex-row gap-6 lg:gap-12">

<!-- If mobile=1 column, desktop=3 columns -->
<div class="grid grid-cols-1 lg:grid-cols-3 gap-6">

<!-- If element hidden on mobile, visible on desktop -->
<div class="hidden lg:block">

Layout Variants

Some sections support multiple layout options independent of breakpoint (e.g., "image left" vs "image right"). Define these as a separate variant property called layout:

13. Images & Assets

Image Layer Naming

Image layers must be named with the img: prefix so the export script knows to extract and reference them via campaign_asset.

Figma Layer NameExported FilenameLiquid Reference
img:hero-bgimages/hero-bg.jpg{{ 'images/hero-bg.jpg' | campaign_asset }}
img:product-bottleimages/product-bottle.png{{ 'images/product-bottle.png' | campaign_asset }}
img:icon-shieldimages/icon-shield.svg{{ 'images/icon-shield.svg' | campaign_asset }}

Image Export Rules

  • Photos/raster images → export as .jpg (quality 80) or .webp
  • Product images with transparency → export as .png
  • Icons and illustrations → export as .svg
  • Background images that are decorative → set via CSS background-image
  • Content images → use <img> tag with alt text from the Figma layer description

Image as Component Property

When an image is dynamic (varies per campaign), expose it as a text property containing the asset path:

// Figma property: background_image (Text) = "images/hero-bg.jpg"
// Liquid output:
<img src="{{ background_image | campaign_asset }}" alt="{{ background_image_alt }}">

Two Image Patterns — Designer Must Choose One

Every section image is either a background or a contained image. The designer declares which by using the correct layer name prefix. Getting this wrong means the export will look broken at responsive widths.

Pattern 1 — bg: Background Pattern 2 — img: Contained
Layer name bg:hero-bg img:hero-product
Design intent Decorative fill, content sits on top, bleeds to edges Discrete visual element alongside content in a column
Figma setup Image fill on the section/column frame. Bleed/offset OK. Must be fully contained within its frame. No bleed. Transparent PNG. Full subject visible.
Code output CSS background-image: cover on the element <img> with object-contain
Asset export export-node.sh on the section/column node export-node.sh on the image layer node

Why not use the raw asset URL from get_design_context? The MCP returns the original uploaded source file — not what Figma renders on canvas. A 1500px wide image cropped with Figma offset still gives you 1500px when downloaded via the raw URL. Always use export-node.sh.

# Always export the node, never use the raw imgRectangle URL
./scripts/export-node.sh {node-id} src/{campaign}/assets/images/{filename}.png 2

14. Liquid Template Conventions

The Three Essential Filters

campaign_asset — Asset Path Resolution

Resolves relative asset paths to the campaign's directory. Every image, CSS, JS, and file reference MUST use this filter.

{{ 'images/hero.jpg' | campaign_asset }}  → /campaign-slug/images/hero.jpg
{{ 'css/global.css' | campaign_asset }}   → /campaign-slug/css/global.css
{{ 'https://cdn.com/img.png' | campaign_asset }} → passes through unchanged

campaign_link — Page URL Generation

Generates clean URLs for inter-page navigation. Every internal link MUST use this filter.

{{ 'checkout.html' | campaign_link }}  → /campaign-slug/checkout/
{{ 'index.html' | campaign_link }}    → /campaign-slug/
{{ '#pricing' | campaign_link }}      → #pricing (anchors pass through)

campaign_include — Partial Rendering

Renders a partial from _includes/ with named arguments. Supports both {{ var }} and {{ include.var }} syntax.

{% campaign_include 'hero-banner.html' headline="Welcome" cta_text="Buy Now" %}
{% campaign_include '_components/cta-button.html' text=btn_text url=btn_url %}

Template Patterns

Conditional Content

{% if show_testimonials %}
  <div class="testimonials-grid">
    <!-- testimonial content -->
  </div>
{% endif %}

Default Values

<h1>{{ headline | default: "Your Headline Here" }}</h1>
<a href="{{ cta_url | default: 'checkout.html' | campaign_link }}">

Iterating Over Items

<!-- For sections with repeating items (testimonials, features, etc.) -->
{% for item in items %}
  {% campaign_include '_components/testimonial-card.html'
     quote=item.quote
     author_name=item.name
     author_image=item.image %}
{% endfor %}

JS Integration Pattern

// assets/js/faq-accordion.js
document.addEventListener('DOMContentLoaded', () => {
  document.querySelectorAll('[data-faq-toggle]').forEach(toggle => {
    toggle.addEventListener('click', () => {
      const item = toggle.closest('[data-faq-item]');
      const answer = item.querySelector('[data-faq-answer]');
      const isOpen = item.getAttribute('data-open') === 'true';
      item.setAttribute('data-open', !isOpen);
      answer.style.maxHeight = isOpen ? '0' : answer.scrollHeight + 'px';
    });
  });
});

The section's Liquid template uses data-* attributes as hooks — never class names:

    <p>{{ answer }}</p>
  </div>
</div>

13. Section Export Process

This is the validated end-to-end process established while exporting the hero-1 section. Follow these steps for every new section.

Step 1 — Fetch all 3 breakpoint nodes in parallel

Every section has desktop, tablet, and mobile variants. Always fetch all three at once with get_design_context before writing any HTML.

# Fetch all three breakpoint nodes simultaneously
get_design_context(nodeId: "143:10518", fileKey: "...")  # desktop
get_design_context(nodeId: "143:10610", fileKey: "...")  # tablet
get_design_context(nodeId: "143:12936", fileKey: "...")  # mobile

Record these values from each variant before writing any code:

  • Desktop: outer padding, inner gap, column widths, font sizes
  • Tablet: outer padding, inner gap, font size deltas from desktop
  • Mobile: stacking order, text alignment, image height, element reordering

Step 2 — Apply the 4-layer HTML structure

Every section must follow this wrapper pattern. No exceptions.

<section class="bg-white w-full overflow-hidden">

  <!-- CONTAINER: max-width cap + Figma outer padding -->
  <div class="max-w-[1440px] mx-auto md:pr-[26px] lg:pr-[120px]">

    <!-- CONTENT: flex row with gap from inner Figma node -->
    <div class="flex flex-col md:flex-row md:items-stretch md:gap-8 lg:gap-[60px]">

      <!-- COMPONENTS: image column, content column -->
      <div class="md:flex-1">image</div>
      <div class="md:flex-1">content</div>

    </div>
  </div>
</section>
Figma NodeHTML LayerKey Classes
Section frame (e.g. 143:10518)<section>bg-* overflow-hidden w-full
Section frame paddingContainer <div>max-w-[1440px] mx-auto md:pr-[N] lg:pr-[N]
Inner flex node (e.g. 143:10519)Content <div>flex flex-col md:flex-row md:gap-[N] lg:gap-[N]
Child column nodesComponent <div>smd:flex-1 per column

Why this structure

The max-w-[1440px] mx-auto container is what actually enforces the width cap — without it, flex-1 columns expand infinitely. The right padding from the Figma section frame sits on this container, not on the content column. The image column has no left padding — it bleeds to the container's left edge, matching the Figma design.

Step 3 — Responsive column rules

Figma pixel widths are 1440px reference values — never hard-code them as column widths.

❌ Wrong✅ CorrectReason
lg:w-[680px] lg:flex-nonemd:flex-1Fixed width stops at 1024px
lg:w-[580px] lg:flex-nonemd:flex-1Fixed width stops at 1024px
w-[440px] on CTA blockmd:max-w-[440px] w-fullUse max-w, not fixed w
w-[475px] on headingRemove entirelyLet text wrap naturally
whitespace-nowrap on bulletsRemoveCauses overflow on narrow viewports

Step 4 — Handle mobile element reordering

When the mobile layout places an element above the image (e.g. heading), but the image comes first in the content column structure, duplicate the element using visibility classes rather than CSS order:

<!-- Mobile only: heading appears above image in DOM -->
<div class="md:hidden flex flex-col items-center text-center px-4 pt-6">
  <h1>{{ headline }}</h1>
</div>

<!-- Image column -->
<div class="relative h-[300px] md:h-auto md:flex-1">...</div>

<!-- Content column: heading hidden on mobile (duplicate above handles it) -->
<div class="md:flex-1">
  <div class="hidden md:flex flex-col">
    <h1>{{ headline }}</h1>
    <p>{{ subheadline }}</p>
  </div>
  <!-- bullets, CTA, testimonial always visible -->
</div>

Step 5 — Save reference screenshots

Immediately after fetching the 3 design contexts, save a reference PNG of each breakpoint. These give Claude an accurate visual target when writing HTML and serve as a permanent record of the design intent.

./scripts/save-ref.sh {campaign} {section} {desktop-id} {tablet-id} {mobile-id}

# Example:
./scripts/save-ref.sh section-preview hero-2 143:10703 143:10748 143:13028

Output (gitignored — local only):

src/{campaign}/_ref/
  hero-2-desktop.png   ← visual target at 1440px
  hero-2-tablet.png    ← visual target at 768px
  hero-2-mobile.png    ← visual target at 375px

Requires .env token

The script uses FIGMA_ACCESS_TOKEN from your .env file to call the Figma REST API export endpoint. See Getting Started Step 2 for setup.

Step 6 — Download Figma assets immediately

Asset URLs from get_design_context expire after 7 days. Download them before previewing.

# Download assets — detect actual format before naming
curl -sL "[check-url]"    -o "assets/images/icon-check.tmp"
curl -sL "[arrow-url]"    -o "assets/images/icon-arrow.tmp"
curl -sL "[photo-url]"    -o "assets/images/hero-photo.tmp"

# Always verify format — Figma returns SVG for vectors
file assets/images/*.tmp
# icon-check.tmp: SVG Scalable Vector Graphics image  → rename .svg
# icon-arrow.tmp: SVG Scalable Vector Graphics image  → rename .svg
# hero-photo.tmp: PNG image data, 1500x647 ...        → rename .png

# Image fills (hero photos, backgrounds) don't surface as URLs.
# Export manually: right-click frame in Figma → Export → PNG

Figma MCP always returns SVG for vector assets

Icons, badges, and illustrations are returned as SVG regardless of what filename extension you use when saving. Saving a vector as .png causes it to silently not render in the browser. Always run file on downloaded assets to verify format and rename accordingly.

Common Mistakes

Missing max-width container

Without max-w-[1440px] mx-auto, flex-1 columns expand infinitely on wide screens. The container div is not optional.

Fixed column widths from Figma

Figma shows w-[680px] and w-[580px] because the canvas is 1440px. These are reference values — translating them directly as lg:w-[680px] lg:flex-none freezes the layout at 1024px and breaks responsiveness above that.

Padding on the wrong element

The right padding from the Figma section frame (e.g. pr-[120px] on desktop) belongs on the container div, not the content column. Putting it on the content column leaves the image column incorrectly padded.

Tailwind CDN in section partials

Tailwind CDN belongs in _layouts/base.html only. Sections are partials — they don't own the document head. Adding it to a section causes it to load multiple times.

Wrong path in campaign_include

The campaign_include tag automatically resolves paths relative to the campaign's _includes/ directory. Prefixing the path with _includes/ doubles it and silently renders nothing.

❌ Wrong✅ Correct
{% campaign_include '_includes/hero.html' %}{% campaign_include 'hero.html' %}
{% campaign_include '_includes/_components/btn.html' %}{% campaign_include '_components/btn.html' %}
